Phases of Computer Program Development

  • Phase 1: Understand the problem
  • Phase 2: Design an algorithm to solve the problem
  • Phase 3: Write the program code in an appropriate language based on the algorithm
  • Phase 4: Enter the program and save
  • Phase 5: Execute the program
  • Phase 6: Test program, find errors and debug, then retest
  • Phase 7: Document and evaluate the program

    Programming Structures

  • Sequential structures: Top-down execution
  • Decision structures: if/then/else

    • Interative (looping):

  • While loop: Continue looping as long as something is true
  • Repeat/until loop: Continue looping until something is true
  • For loop: Loop a given number of times

    • Recursive structures

  • Repeat a set of instructions
  • Repetition continues until:
  • Degenerative case (also known as a "base case"), which is a termination condition, is met

    • Two classic programming/database problems:

    Sorting: Descending/ascending order:

  • Sorting numbers
  • Sorting strings (in accordance with ASCII values)

    • Searching: Find specific data that meets a certain criteria

    In database:
  • Filters: Temporary search
  • Queries: Serch criteria are saved so they don't have to be recreated
    • Two popular search techniques:
  • Sequential search: Search from top to bottom or vice versa
  • Binary search: (Only works if data is sorted first) Starts in the middle, then go half of that, etc.

    • Two classic program goals:

  • Efficency: Program takes minimal time and/or resources
  • Correctness: Program runs as expected and works for all expected conditions

    • Software Testing and Verifications

    Kind of tests:

  • Acceptance testing: Verifying if the whole system works as intended
  • Integration testing: Verifying that software components and functions work together
  • Unit testing:Validating that each software unit performs as expected (a unit is the smallest testable component of an application)
  • Functional testing:Checking functions by emulating business scenarios, based on functional requirements
  • Performance testing: Testing how the software works under different workloads
  • Regression testing: Checking whether or not new features break or degrade functionally
  • Stress testing: Testing how much strain the system can take before it fails
  • Usability testing: Validating how well a customer can use the system to complete a task

    • Software testing is done to detect:

  • Arcitechtual flaws
  • Poor design decisions
  • Invalid or incorrect functionality
  • Security vulnerabilities

    • Best testing pracvtices:

  • Continuous testing: Project teams test each build as it becomes available
  • Configuration management: Organizations centrally maintain test assets and track what software builds to test
  • Service visualization: Simulates the services and systems that are missing or not yet completed, enabling teams to reduce dependencies and test sooner
  • Defect or "bug tracking": Automated tools allow teams to track defects, measure their scope and impact, and uncover related issues
  • Metrics and reporting: Allows teams to track issuesReporting and analytics enable team members to share status, goals and test results
